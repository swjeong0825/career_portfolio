About

A backend engineer who builds backend services and data pipelines that scale, evolve cleanly, and fail predictably.

Most of my Amazon work has lived where ambiguity meets scale — designing event-driven pipelines on SQS and Lambda, building contract-first services on API Gateway with Smithy and AWS CDK, and shipping ML/LLM workflows that operate against 600M-item catalogs. One project I'm particularly proud of: I built the team's first real-time Customer Affinity pipeline (SQS → Lambda → DynamoDB) to replace a daily batch job with sub-minute signal freshness, then validated the end-to-end system in a four-week production A/B test that showed a ~5% directional lift in click rate. The read API was later adopted by another team's production use case. I care about systems that fail predictably, evolve cleanly, and stay observable in production.

On personal projects I've leaned into LLM-powered UX. The pattern I keep coming back to: put the LLM in front of users for natural-language input, but keep it out of the write path. In the Tennis League Management Bot, the chat-to-intent service interprets user messages and returns either read data or a pre-filled form payload — the user always confirms, and domain validation lives only in the core backend.

I'm comfortable across Python, Java, and TypeScript, with deep AWS experience and growing depth in DDD-style backend modeling and structured-output LLM design. Currently exploring roles where backend rigor and LLM-driven product surfaces meet.
