Tennis League Management Bot

A tennis match-recording app for casual groups. The whole product is a single chat box backed by an LLM agent — users type things like "record match result" or "show standings" and the agent guides them through. The interesting part isn't the chat; it's the backend architecture that keeps the LLM useful without letting it write to the database.

The architectural problem

A simple chat UI shifts complexity to the backend.

Many existing tennis-league apps require players to learn a new interface, navigate multiple screens, and understand specific workflows before they can use the product effectively. I wanted to reduce that friction as much as possible, so the entire product is a single chat box powered by an AI agent.

That simple frontend creates an interesting backend problem. Instead of putting most of the complexity in the UI, the system shifts it to the backend architecture. The interesting question becomes: how do you let an LLM be the front door without letting it become the source of truth, and without bypassing domain validation?

System architecture

Three services, one boundary that matters.

The app is split into a vanilla static frontend, an LLM-powered chat-to-intent service, and a core domain backend. The boundary that matters is between the chat service and writes — the chat service never issues write requests.

Frontend (Browser) | | POST /chat { client_message, last_server_message, league_id, [host_token] } v Chat-to-Intent Server ---- LLM (intent + parameter extraction) | | GET only v TLM Backend Main ---- PostgreSQL (domain data) ^ | Confirmed write forms (POST / PATCH / DELETE) | go DIRECTLY from frontend, bypassing chat service Frontend (Browser)
Key constraint: For read operations (standings, history, roster), the chat-to-intent service calls backend GET endpoints and returns a user-friendly response. For writes (submit match, edit nickname, delete record), it returns a pre-filled form payload only. The frontend renders that payload, the user confirms or edits it, and submits the final request directly to the backend. Human confirmation stays in the loop, and the LLM layer never mutates production data.

Subsystems

Three repositories with clear ownership.

Each piece is small enough to reason about in isolation. The frontend owns presentation and confirmation; the chat service owns intent classification and parameter extraction; the backend owns persistence and domain validation.

TLMB Backend Main

  • FastAPI
  • PostgreSQL
  • SQLAlchemy
  • Alembic
  • DDD

Domain logic, persistence, and the public REST API. Modeled with Domain-Driven Design: League and Match as aggregate roots with value objects, policies, and domain services. Layered (domain / application / infrastructure) with a Repository pattern and Unit of Work for transactional consistency. Admin operations are gated by an X-Host-Token header.

View repository

TLMB Chat-to-Intent

  • FastAPI
  • OpenAI
  • Groq
  • Gemini
  • Stateless

An LLM-powered intermediary classifying natural-language messages into 10 intents. Two-stage pipeline: intent classification with per-intent confidence thresholds, then parameter extraction. Low-confidence inputs return a structured clarification question. Stateless — the client carries context via last_server_message. Owns no domain logic; reads from the backend via GET only.

TLMB Frontend

  • HTML
  • CSS
  • Vanilla JS
  • i18n (en/ko)

No-build static client — plain HTML/CSS/JS, no framework or bundler. Each page is its own index.html plus a per-page entry script and a small set of shared helpers (i18n, config, error mapping). When the chat returns a write intent, the browser renders the pre-filled form for the user to confirm and submits directly to the backend.

Infra & dev practice

Isolated environments for AI-assisted development.

The backend is hosted on Railway and the database uses Neon serverless Postgres. Because I lean heavily on AI agents during development, my main infrastructure requirement was safe, isolated environments for feature work. Railway makes it easy to spin up a separate backend environment per development branch, while Neon's database branching gives each branch an isolated copy of the database.

That setup lets AI agents implement and test features more freely — running real integration tests against real data — without risking any change to production. It's a small infra decision that has an outsized effect on how fast I can iterate.
